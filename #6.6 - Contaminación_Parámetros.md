# Contaminación de Parámetros
Las vulnerabilidades de contaminación de parámetros HTTP (HPP) pueden introducirse si múltiples parámetros HTTP tienen el mismo nombre. Este problema puede hacer que una aplicación interprete los valores incorrectamente. Un atacante puede explotar las vulnerabilidades de HPP para eludir la validación de entrada, provocar errores en la aplicación o modificar valores de variables internas.<br>
__NOTA__: Las vulnerabilidades de HPP pueden provocar ataques del lado del servidor y del cliente.<br>

Un atacante puede encontrar vulnerabilidades de HPP buscando formularios o acciones que permitan la entrada proporcionada por el usuario. Luego, el atacante puede agregar el mismo parámetro a los datos __GET__ o __POST__, pero con un valor diferente:<br>

```
    https://store.h4cker.org/?search=cars&results=20
```

Esta URL tiene la cadena de consulta __search__ y el valor de parámetro __cars__. El parámetro puede estar oculto entre otros varios parámetros. Un atacante podría dejar el parámetro actual y agregar un duplicado, como se muestra aquí:<br>

```
    https://store.h4cker.org/?search=cars&results=20&search=bikes
```
Después de enviar la solicitud, el atacante podría analizar la página de respuesta para identificar si la aplicación analizó alguno de los valores ingresados. A veces es necesario enviar tres solicitudes HTTP para cada parámetro HTTP. Si la respuesta del tercer parámetro es diferente a la del primero y del segundo, esto puede ser un indicador de una falta de coincidencia de impedancia que podría abusarse para desencadenar vulnerabilidades en HPP.<br>

# Vulnerabilidades de Referencias Directas Inseguras a Objetos
Las vulnerabilidades de Referencias Directas Inseguras a Objetos se pueden explotar cuando las aplicaciones web permiten el acceso directo a los objetos en función de la entrada del usuario. Una explotación exitosa podría permitir que los atacantes eludan la autorización y accedan a recursos que el sistema debe proteger. Este tipo de vulnerabilidad se produce cuando una aplicación no sanitiza la entrada del usuario y no realiza las comprobaciones de seguridad adecuadas.<br>

Un atacante puede aprovechar las vulnerabilidades de las Referencias Directas Inseguras a Objetos modificando el valor de un parámetro utilizado para apuntar directamente a un objeto. Para explotar este tipo de vulnerabilidad, un atacante debe trazar un mapa de todas las ubicaciones de la aplicación donde se utiliza la entrada del usuario para hacer referencia a los objetos directamente.<br>

El siguiente ejemplo muestra cómo el valor de un parámetro se puede utilizar directamente para recuperar un registro de la base de datos:<br>

```
    https://store.h4cker.org/changepassd?user=omar
```

En este ejemplo, el valor del parámetro de usuario se usa para que el sistema cambie la contraseña del usuario. Un atacante puede probar con otros nombres de usuario y ver si es posible modificar la contraseña de otro usuario.<br>

__CONSEJO__: Las mitigaciones para este tipo de vulnerabilidad incluyen la validación de entrada, el uso de referencias de objetos indirectos por usuario o sesión, y verificaciones de control de acceso para garantizar que el usuario esté autorizado para el objeto solicitado.<br>
