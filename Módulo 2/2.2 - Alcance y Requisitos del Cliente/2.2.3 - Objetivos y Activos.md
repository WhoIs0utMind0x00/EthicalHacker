# 2.2.3 - Lista de Objetivos y Activos dentro del alcance
El alcance es uno de los elementos más importantes de las tareas previas al compromiso con cualquier participación en una prueba de penetración. No solo debe identificar y documentar cuidadosamente todos los sistemas, aplicaciones y redes que se probarán, sino también determinar los requisitos y calificaciones específicos necesarios para realizar la prueba. Cuanto más amplio sea el compromiso de las pruebas de penetración, más habilidades y requisitos se necesitarán.

Su alcance y la documentación relacionada deben incluir información sobre los tipos de redes que se probarán. En la tabla 2-3, vio algunos ejemplos de los rangos de direcciones IP de los dispositivos y activos que el evaluador de penetración puede evaluar. Además de los rangos de IP, debe documentar las redes inalámbricas o los identificadores de conjuntos de servicios (SSID) que puede o no probar.

También es posible que lo contraten para realizar una evaluación de aplicaciones modernas mediante diferentes interfaces de programación de aplicaciones (API).<br>

### SOAP
Archivos de proyecto del Protocolo Pimple de Acceso a Objetos (SOAP): SOAP es un estándar de API que se basa en XML y esquemas relacionados. Las especificaciones basadas en XML se rigen por documentos de definición de esquema XML (XSD). Tener una buena referencia de lo que admite una API específica puede ser muy beneficioso para un evaluador de penetración y acelerará las pruebas. Se puede acceder a la especificación SOAP en https://www.w3.org/TR/soap.
#
### Swagger
La documentación de Swagger (OpenAPI) es un marco moderno de documentación y desarrollo de API que ahora es la base de la especificación de OpenAPI (OAS). Estos documentos se utilizan en las API de transferencia de estado representacional (REST). REST es un estilo de arquitectura de software diseñado para guiar el desarrollo de la arquitectura de servicios web (incluidas las API). Las API REST o “RESTful” son los tipos más comunes de API que se utilizan en la actualidad. Los documentos de Swagger pueden ser extremadamente beneficiosos al probar las API. Puede obtener información adicional sobre Swagger en https://swagger.io. OAS está disponible en https://github.com/OAI/OpenAPI-Specification.
#
### WSDL
El lenguaje de descripción de servicios web (WSDL) es un lenguaje basado en XML que se utiliza para documentar la funcionalidad de un servicio web. Se puede acceder a la especificación WSDL en https://www.w3.org/TR/wsdl20-primer.
#
### GraphQL
GraphQL es un lenguaje de consulta para API. También es un tiempo de ejecución del lado del servidor para ejecutar consultas mediante un sistema de tipos que defina para sus datos. Se puede acceder a información técnica adicional sobre GraphQL en https://graphql.org/learn.
#
### WADL
El lenguaje de descripción de aplicaciones web (WADL) es un lenguaje basado en XML para describir aplicaciones web. La especificación WADL se puede obtener en https://www.w3.org/Submission/wadl.<br>

Los siguientes son algunos recursos de soporte adicionales que puede obtener de la organización que lo contrató para realizar la prueba de penetración.<br>

### Kit de Desarrollo de Software (SDK) para Aplicaciones Específicas
Un SDK, o devkit, es un conjunto de herramientas de desarrollo de software que se pueden utilizar para interactuar e implementar un marco de software, un sistema operativo o una plataforma de hardware. Los SDK también pueden ayudar a los evaluadores de penetración a comprender ciertas aplicaciones especializadas y plataformas de hardware dentro de la organización que se está probando.
#
### Acceso al Código Fuente
Algunas organizaciones pueden permitirle obtener acceso al código fuente de las aplicaciones que se probarán.
#
### Ejemplos de Solicitudes de Aplicación
En la mayoría de los casos, podrá revelar el contexto mediante el uso de herramientas de prueba de aplicaciones web, como proxies como Burp Suite y OWASP Zed Attack Proxy (ZAP). Obtendrá más información sobre estas herramientas en el Módulo 6, “Aprovechamiento de las vulnerabilidades basadas en aplicaciones”, y en el Módulo 10, “Herramientas y análisis de código”.
#
### Diagramas de Arquitectura de Sistemas y Redes
Estos documentos pueden ser muy beneficiosos para los evaluadores de penetración y pueden usarse para documentar y definir qué sistemas están dentro del alcance durante las pruebas.<br>
<hr>
Es muy importante documentar la ubicación física donde se realizarán las pruebas de penetración, así como los nombres de dominio totalmente calificados (FQDN) del Sistema de nombres de dominio (DNS) de las aplicaciones y los recursos permitidos (incluidos los subdominios). También debe aceptar y comprender si se le permitirá demostrar cómo un atacante externo podría poner en riesgo sus sistemas o cómo un agente interno podría comprometer los activos internos. Esta identificación y alcance de objetivos externos vs. internos deben documentarse claramente.

En el Módulo 1, aprendió que hay varias consideraciones ambientales, como las aplicaciones alojadas en una nube pública. Comprender el concepto de aplicaciones alojadas de origen frente a aplicaciones alojadas de terceros es muy importante. Las aplicaciones actuales no solo se pueden alojar en una nube pública (como AWS, GCP o Azure), sino también en nubes privadas e híbridas. Como evaluador de penetración, debe familiarizarse con las restricciones y limitaciones impuestas por los proveedores de alojamiento o nube de terceros.

Aumento del alcance (Scope Creep) es un término de administración de proyectos que se refiere al crecimiento descontrolado del alcance de un proyecto. También se suele denominar síndrome del fregadero de la cocina, aumento de los requisitos y aumento de las funciones. El aumento del alcance puede sacarlo del negocio. Muchas empresas de seguridad sufren aumentos en el alcance y no tienen éxito porque no tienen idea de cómo identificar cuándo comienza el problema o cómo reaccionar ante él.

#
#### Elementos Técnicos y No Técnicos
Cuando no haya una identificación eficaz de qué elementos técnicos y no técnicos se requerirán para la prueba de penetración.

#### Mala Comunicación
Cuando hay poca comunicación entre las partes interesadas, incluidos su cliente y su propio equipo.
